# MyBatis SQL 작성 가이드

> **적용 대상:** `**/*Mapper.xml` (MyBatis를 사용하는 모든 프로젝트)
> **DB:** `{{config.db.vendor}}` 스키마 `{{config.db.schema}}`
> **단일 출처:** `.claude/config/workspace.yaml` `db.*`

mapper XML(`*Mapper.xml`) 작성 시 **Part A. 작성 규칙**을 준수하고, **Part B. 분석 지침**으로 처음부터 최적화된 SQL을 작성한다. 신규 인덱스 추가가 필요한 경우 **Part C** 패턴을 따른다.

## 2. SQL 전체 대문자

SQL 문은 키워드 / 테이블명 / 컬럼명 모두 대문자로 작성한다.

| 항목     | 예시                                                         |
| -------- | ------------------------------------------------------------ |
| 키워드   | `SELECT`, `FROM`, `WHERE`, `JOIN`, `ORDER BY`, `GROUP BY` 등 |
| 테이블명 | `TB_USER`, `TB_PAY_LDGR` 등                                  |
| 컬럼명   | `USER_ID`, `USER_NM`, `REG_DT` 등                            |

**예외 (camelCase 유지):**

- MyBatis 파라미터: `#{paramName}`
- 동적 태그(`<if>`, `<foreach>` 등) 내 변수명

```xml
<select id="selectUser" resultType="UserDto">
    SELECT USER_ID, USER_NM, EMAIL
    FROM TB_USER
    WHERE USER_ID = #{userId}
        <if test="email != null">
            AND EMAIL = #{email}
        </if>
</select>
```

---

## 3. `SELECT *` 금지

필요한 컬럼을 명시적으로 나열한다.

```xml
<!-- ❌ 금지 -->
SELECT * FROM TB_USER

<!-- ✅ 허용 -->
SELECT USER_ID, USER_NM, EMAIL FROM TB_USER
```

## Part B. 분석 지침 (최적화된 SQL 작성용)

## 4. 작성 시 자가 점검 항목

mapper에서 SQL 작성/수정 시 다음 8개 항목을 점검한다.

| 항목             | 점검 내용                                                                             |
| ---------------- | ------------------------------------------------------------------------------------- |
| 쿼리 목적        | 비즈니스 목적, 호출 위치, 실행 빈도                                                   |
| 입/출력          | 파라미터 → SQL 컬럼 매핑, 결과 컬럼 목록                                              |
| JOIN 구조        | 각 JOIN의 목적, 테이블 간 관계, 드라이빙 테이블                                       |
| WHERE 조건       | 각 조건이 사용하는 인덱스 (PK / UNIQUE / 일반 / 없음)                                 |
| 동적 SQL         | `<if>`/`<choose>`/`<foreach>`/`<where>`/`<set>`/`<trim>` 분기별 인덱스 활용 (§5 참조) |
| 사용자 정의 함수 | `FN_*` 함수의 호출 위치/빈도/내부 I/O (§6 참조)                                       |
| 인덱스 활용      | 각 접근 방식 평가 (Eq / Range / Seq Scan) (§7 참조)                                   |
| 최적화 포인트    | JOIN 순서, 인덱스 미활용, 불필요 JOIN, 중복 조건, N+1 가능성                          |

## 5. 동적 SQL 분기별 인덱스 활용

MyBatis 동적 SQL은 조건 분기에 따라 실행 SQL이 달라지므로, 각 분기별로 인덱스 활용을 별도로 분석한다.

- **`<if>` 태그:** 조건이 true/false일 때 각각의 WHERE 절 구성과 인덱스 활용 차이를 분석한다. 특히 **선택적 조건이 빠졌을 때 인덱스 활용이 불가능해지는 케이스**에 주의한다.
- **`<choose>`/`<when>`/`<otherwise>`:** 각 분기별 실행 경로에서 사용하는 조건과 인덱스를 비교한다.
- **`<foreach>`:** IN 절 바인드 변수 개수에 따른 성능 영향을 분석한다. **대량 리스트(100건 이상)는 임시 테이블 JOIN 전환**을 고려한다.
- **`<where>`/`<set>`/`<trim>`:** 자동 생성되는 WHERE/SET 절의 최종 형태를 파악하여 인덱스 활용을 분석한다.

---

## 6. 사용자 정의 함수(FN\_\*) 호출 시 고려 사항

SQL 내 사용자 정의 함수는 쿼리 성능에 큰 영향을 줄 수 있으므로 다음을 점검한다.

| 점검 항목   | 설명                                                                                                    |
| ----------- | ------------------------------------------------------------------------------------------------------- |
| 호출 위치   | SELECT 절 / WHERE 절 / JOIN 조건 — **WHERE 절 호출 시 인덱스 활용 차단 가능, 영향 큼**                  |
| 호출 빈도   | 행(row)당 1회 / 그룹(GROUP BY)당 1회 — 빈도가 높을수록 성능 영향 증가                                   |
| 내부 I/O    | 함수가 다른 테이블을 조회하는지, 어떤 인덱스를 사용하는지 (함수 소스 미확인 시 "내부 로직 미확인" 표시) |
| 캐싱 가능성 | 동일 입력 반복 호출 시 DB 엔진 내부 캐싱 또는 애플리케이션 캐시 적용 가능 여부                          |

---

## 7. 인덱스 활용 패턴

| 접근 방식    | 의미                                                  | 활용                                             |
| ------------ | ----------------------------------------------------- | ------------------------------------------------ |
| **Eq**       | `WHERE col = value` — 인덱스 정확 매치                | 가장 빠름, PK / UNIQUE / 일반 인덱스 모두 효과적 |
| **Range**    | `WHERE col >= a AND col <= b`, `BETWEEN`, `LIKE 'x%'` | 인덱스 정렬을 활용한 범위 스캔                   |
| **Seq Scan** | 인덱스 미활용, 전체 테이블 순차 스캔                  | 작은 테이블·소량 행 한정. 큰 테이블에서는 회피   |

## Part C. 신규 인덱스 제안

## 8. CREATE INDEX CONCURRENTLY 패턴

기존 쿼리 수정만으로 효과가 제한적이고 신규 인덱스 추가가 필요한 경우:

```sql
-- 운영 환경: CONCURRENTLY 옵션으로 테이블 잠금 없이 인덱스 생성
CREATE INDEX CONCURRENTLY {인덱스명} ON {테이블명} ({컬럼 목록});
```

**제안 시 포함할 정보:**

- 인덱스명 (예: `idx_tb_user_email_dept`)
- 대상 컬럼 (단일 / 복합)
- 제안 이유 (어떤 쿼리에서 어떻게 활용)
- 예상 효과 (Seq Scan → Eq/Range, 응답 시간 등)

> 일반 `CREATE INDEX`는 테이블에 `ACCESS EXCLUSIVE` 락을 걸어 운영 영향이 크다. 운영 환경에서는 반드시 `CONCURRENTLY` 옵션을 사용한다.
